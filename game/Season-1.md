# Первый сезон: Первая синхронизация

## Цель сезона

Пользователь подключается к Odyssey, знакомится с экипажем, проходит диагностику и восстанавливает сектор `Agent Systems` на подходящем уровне глубины.

Сезон доказывает основную идею OrbitQuest: знание считается стабильным только после практического доказательства, а его актуальность требует постоянного обслуживания.

## Пролог. Подключение

Полная последовательность описана в [[game/Prologue]].

1. входящее приглашение Atlas;
2. активация Odyssey;
3. знакомство с KORA;
4. демонстрация Atlas и Сдвига;
5. подтверждённый сигнал VEGA;
6. демонстрация PIX;
7. объяснение роли Навигатора;
8. диагностика и раскрытие маршрута.

Пролог не начинается с `TOOLS-03`: миссия появляется только после знакомства и placement.

## Три маршрута сезона

Темы сектора общие, но вход, scaffolding и финальный критерий различаются. Канон маршрутов: [[learning/Placement-and-Routes]].

### Освоение

**Сюжет:** Навигатор активирует базовые системы Odyssey и постепенно восстанавливает понимание Atlas.

**Главы:**

1. модель, инструкции и контекст;
2. structured output и проверяемый результат;
3. tools и границы разрешений;
4. API, ошибки и повторные попытки;
5. MCP и внешние системы;
6. память и приватность;
7. tests, evals и доказательство.

**Финал:** собрать небольшого агента по готовой архитектурной схеме, самостоятельно заполнить критические участки и пройти tests.

### Практика

**Сюжет:** базовые системы работают, но Сдвиг повредил несколько сервисных модулей. Пользователь становится инженером восстановления.

**Главы:**

1. найти ошибку в контексте и критериях;
2. исправить невалидный structured output;
3. восстановить tool handler и permission boundary;
4. починить retries, timeout и rate limits;
5. отладить MCP client/server interaction;
6. исправить state и memory leak;
7. построить tests и evals для переноса навыка.

**Финал:** отремонтировать и защитить сломанного агента без пошаговой схемы.

### Актуализация

**Сюжет:** системы стабильны. Навигатор назначается дежурным аудитором Atlas и работает с реальными изменениями технологий.

**Структура не линейная:**

- подтверждённые releases;
- API migrations;
- deprecated-подходы;
- security advisories;
- новые production-рекомендации;
- изменения MCP и agent tooling;
- проверка freshness старых решений.

**Финал:** провести аудит и миграцию существующего агента после реального обновления, сохранив tests, security boundary и доказательства источников.

## Общие поворотные точки

Независимо от маршрута пользователь встречает одинаковые события мира:

1. **Первый стабильный спутник** — обучение меняет мир только после evidence.
2. **Ложный сигнал** — VEGA показывает отличие источника от marketing claim.
3. **Красный симулятор** — PIX демонстрирует, что уверенное объяснение не заменяет tests.
4. **Неуверенность KORA** — наставник прямо признаёт недостаточность данных.
5. **Конфликт источников** — решение откладывается или публикуется как uncertain.
6. **Перенос навыка** — знакомый принцип появляется в новом контексте.
7. **Финальная синхронизация** — результат становится реальным артефактом.

## Финальное открытие

После восстановления сектора VEGA сообщает, что новые изменения уже появились в другом участке Atlas. Сдвиг не был аварией, которую можно завершить одной кампанией. Odyssey становится постоянной платформой исследования и обслуживания знаний.

Это открывает live-service цикл без искусственного обнуления прогресса.

## Narrative pacing

- знакомство: 3–4 минуты один раз;
- route reveal: до 45 секунд;
- брифинг обычной миссии: 20–40 секунд;
- delta-брифинг: до 60 секунд с источником;
- сцены между главами: до 2 минут и пропускаются;
- учебный смысл доступен без анимации и звука;
- повторный вход сразу возвращает к текущей миссии.
