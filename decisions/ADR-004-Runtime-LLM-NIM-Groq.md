# ADR-004: Runtime-LLM только через kora-worker (NIM → Groq)

## Контекст

ADR-002 закреплял «только Gemini API в runtime». Это стало неисполнимым:

- Gemini геоблокирует запросы с IP владельца (`FAILED_PRECONDITION: User location is not supported`);
- основной API-ключ Gemini заблокирован по биллингу (limit 0 на всех моделях);
- в клиенте PWA ключи хранить нельзя в принципе — GitHub Pages раздаёт статику публично.

При этом v1 нуждается только в одной живой функции: свободные ответы KORA на вопрос игрока.
Скоринг, маршрут, capability — детерминированные по канону [[learning/Placement-and-Routes]],
LLM туда не допускается.

## Решение

1. Единственная точка runtime-LLM — Cloudflare Worker `apps/kora-worker` (`POST /ask`).
2. Каскад провайдеров: NVIDIA NIM (primary) → Groq (fallback); оба бесплатного tier,
   OpenAI-совместимый chat/completions. Модели задаются vars воркера, смена провайдера —
   правка конфига без деплоя клиента.
3. Ключи живут только в секретах воркера. Клиент знает URL воркера и клиентский
   заголовок-секрет (извлекаем из бандла — принятый остаточный риск: худший случай
   сжигает дневной бюджет, KORA уходит в фолбэк).
4. Недоступность воркера — штатный режим: клиент показывает «Бортовая справка · KORA офлайн»
   из контента миссии. Игра обязана быть полностью проходимой без LLM.
5. LLM по-прежнему не выносит вердиктов, не меняет mastery и не назначает маршрут —
   только объясняет (ограничения вшиты в системный промпт + hintStage).

## Последствия

- Gemini полностью выведен из runtime (ADR-002 отменён). Gemini CLI остаётся допустимым
  инструментом разработки локально.
- Появилась инфраструктурная единица (worker) с ручным деплоем владельца — задокументирован
  в `apps/kora-worker/README.md`; CI-деплой воркера сознательно не строим для v1.
- Лимиты: дневной бюджет KV (200/день), per-IP 40/день, circuit breaker 10 минут после
  5 ошибок подряд — при одном игроке этого достаточно с большим запасом.
